# QEC：E 与 C 辨析与 Code 质量检查

## 何时调用

- 审校一批 QEC JSON/表格，怀疑 E、C 边界混乱
- 新同事第一次写 Explanation
- 合并 codebook 前做一致性检查
- **LLM 批量编码后的质量抽查**（尤其检查 Code 泛化）

## 执行步骤

1. 读取项目中的方法论母本：`methodology-kit/E_vs_C.md`（或与该文档同版本的副本）。
2. 对每条有争议的 QEC，依次问：
   - **删掉 Q 后，E 是否仍是一句完整主张？** 若否，重写 E。
   - **C 是否能在 codebook 中唯一对应？** 若否，改 C 或拆 Q。
   - **读完 C 是否知道具体发生了什么？** 若否（如 C =「关联-功能与场景」），说明 C 泛化，需具体化。参考 `E_vs_C.md` 第 3 节「Code 的具体性标准」。
3. **Code 泛化检查**（LLM 编码必做）：统计唯一率（唯一 Code 数 / 总 Code 数）。若 <90%，说明存在系统性泛化问题，需在 prompt 中增加好/差 Code 对比例并重跑。
4. 仍无法达成一致 → 要求记录 **ADR**（使用 `methodology-kit/ADR_TEMPLATE.md`），不要口头定论。

## 禁止

- 用 C 字段写长句解释
- E 与 C 字面完全同义而不增加信息
- C 使用 Category 名或泛化中间层（如「功能与场景」「需求与痛点」）——必须具体到读出即知行为

## 输出格式

对每条问题编码，按以下结构输出：

```
### 问题 QEC #{编号}
- **原文摘要**：Q 的前 30 字 + ...
- **问题类型**：[E 重复 Q / C 过长 / C 泛化 / E-C 同义 / 其他]
- **具体问题**：一句话说明为什么不合格
- **修改建议**：给出改后的 E 和 C（不需要重写 Q）
```

若为批量检查，在逐条输出后追加一段**汇总**：
- 总检查条数 / 问题条数
- 问题类型分布（如：泛化 12 条、E-C 同义 5 条）
- 唯一率（如适用）
- 整体建议（如：需重跑、需修订 prompt、需补充 codebook 词条）
