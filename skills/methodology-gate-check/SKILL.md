# 方法论阶段闸门

## 何时调用

- Phase 结束准备进入下一阶段
- 对外发报告或论文前
- 内部复盘「能不能让别人复现」

## 检查清单

| 原则 | 自检问题 | 通过 |
|------|----------|------|
| 可追溯 | 每条用于结论的 Q 能否回到原始文件/行或 turn？ | ☐ |
| 可复制 | temperature、模型、prompt 版本是否记录？脚本是否一条命令可跑？ | ☐ |
| 溯因式 | Category/先验框与 Code/主题的边界是否写在 codebook 或 ADR？ | ☐ |
| 适配性 | 编码单位、Category 结构、Code 粒度等是否针对实际数据形态做了调整，并在 ADR 留痕？ | ☐ |
| Code 具体性 | Code 唯一率是否 >90%？是否存在 Category 名复述或泛化中间层？ | ☐ |
| 质量传导 | 数据源质量分级是否影响下游权重或表述？ | ☐ |
| 诚实标注 | 方法边界、未做的一致性检验、样本筛选含义是否在报告或 ADR 中声明？ | ☐ |

### 按阶段聚焦

不同阶段的重点检查项不同。**标记为「必过」的项不通过则不可进入下一阶段**。

| 阶段 | 必过项 | 建议检查项 | 可暂缓项 |
|------|--------|-----------|----------|
| Step 0→1 数据准备完成 | 可追溯 | 适配性 | 质量传导 |
| Step 2→3 编码完成 | 可追溯、Code 具体性 | 可复制、溯因式 | 诚实标注 |
| Step 3→4 聚类完成 | 可复制、适配性 | 溯因式 | — |
| Step 4 发报告前 | **全部 7 项** | — | — |

## 执行步骤

1. 指明当前阶段（例：QEC 完成 / 主题合并完成）。
2. 按上表「按阶段聚焦」确定本次的必过项和建议检查项。
3. 逐项检查；必过项任一项「否」→ **禁止推进**，列出必须补的产出（文档或脚本），设截止日期。
4. 建议检查项「否」→ 记录缺口，允许推进但须在下一阶段补齐。
5. 重大缺口 → 写 ADR 说明「为何本阶段放行」或「为何延期」。
