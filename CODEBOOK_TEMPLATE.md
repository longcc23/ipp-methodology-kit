## 2. QEC 字段定义（与全项目一致）

| 字段 | 定义 | 约束 |
|------|------|------|
| **Q (Quote)** | 可追溯的原文片段 | 必须能定位回原始文件/行或 turn |
| **E (Explanation)** | 研究者对该片段的**解释**：说明「这句话在支持什么判断、机制或态度」 | 可完整句子；允许因果链；**不是**代码标签本身。**若为对话/交互数据**：E 宜显式引用**前序发言**以锚定语境（见 `E_vs_C.md` §6）。 |
| **C (Code)** | **受控标签**，用于统计与主题聚合 | 来自本 codebook 第五节词表，或按「新增代码规则」临时新增并事后合并。**若为对话数据**：同一 turn 可对应 **0～多个** Code，须在本表或 ADR 说明聚合与去重规则。 |

## 6. Code 具体性标准（必须明确）

> 本节确保 Code 在聚合时有区分力。实践经验表明：泛化的 Code 是 LLM 辅助质性编码中最常见的隐性质量问题。

**标准**：读 Code 就知道发生了什么，不需要回看原文。

| 维度 | 说明 |
|------|------|
| 推荐格式 | `{动作}-{具体对象/场景}`，3–8 个汉字 |
| 禁止 | Category 名直接做 Code（如「探询」「关联」） |
| 禁止 | 万能中间层（如「功能与场景」「需求与痛点」） |
| 质量指标 | 唯一率 >90%（唯一 Code 数 / 总 Code 数）；过低说明泛化严重 |
