# QEC 中 E 与 C 怎么分：操作定义与自检

> 目标：减少「E 写成小作文版 Code」和「C 里塞满解释」两类错误。  
> 理论背景：结构化证据单元（Quote–Explanation–Code）常见于可追溯质性分析；**E** 承载推断，**C** 承载可聚合标签。

---

## 1. 一句话区分

| 字段 | 一句话 | 通过测试 |
|------|--------|----------|
| **E (Explanation)** | 这句话**让我们相信什么**（现象、机制、态度、行为）？ | 删掉 Q 之后，E 仍能单独读成一句**可辩驳的主张** |
| **C (Code)** | 在 codebook 里**贴哪个标准标签**？ | 在词表里能**唯一或优先**对应到一条；像「档案柜抽屉名」 |

---

## 2. 自检句（编码时心里念一遍）

1. **E 是否只是在重复 Q？** → 若是，要写出「从 Q 推出的那一步」是什么。  
2. **C 是否超过 8～12 个汉字仍像一句话？** → 可能把解释塞进了 C，应挪到 E。  
3. **同一个 C 能否贴给多篇不相似的 Q？** → 若不能，C 太细或 E 不足；需要改词表或拆 Q。  
4. **E 里是否出现「属于某某类」而 C 为空？** → 把「某某类」收成真正的 Code 名写入 C。

---

## 3. 正例（结构示意）

| Q（节选） | E | C |
|-----------|---|---|
| 「我先问他平时拍照多不多……」 | 销售通过**提问**收集使用场景信息，为后续匹配机型做依据 | `探询-使用场景` |
| 「差 300 块我帮他从以旧换新找回来」 | 面对价格敏感，通过**组合权益**维持成交价感知，而非单纯降价 | `异议处理-价值重构` |

---

## 4. 反例与改法

| 问题 | 原样 | 改法 |
|------|------|------|
| C 里写满机制 | C = 「通过提问了解需求再推荐」 | C 收窄为词表标签；机制全文放 E |
| E 只有「客户提到价格」 | 过薄 | E 补全：「提及价格」**意味着什么**（敏感/比价/预算硬上限） |
| E 与 C 完全同义 | E = 「价格异议」，C = 「价格异议」 | E 写**如何回应或证据强度**；C 保持类别名 |

---

## 5. 与 Stream 1A / 其他数据源的特别提示

- **对话数据**：E 宜写「**该轮在对话功能上承担的角色**」（信息收集 / 推进成交 / 修复关系），避免复述说话内容。  
- **评价文本**：一条评价可拆多条 QEC；**一条 Q 不宜混合两个独立抱怨**。  
- **高分样本子集**：E 中可注明「该条来自高分筛选子样本」，便于 Cross-stream 对照。

---

## 6. 需要升级讨论时

若团队对同一批 Q 的 E/C 仍争执不下：

1. 先检查 **codebook 该 Code 的包含/排除** 是否写清；  
2. 仍不行 → 开一条 **ADR**，允许暂时双码或新增第三码，并设合并截止日。
