# QEC 中 E 与 C 怎么分：操作定义与自检

> 目标：减少「E 写成小作文版 Code」和「C 里塞满解释」两类错误，以及「C 写成泛化标签」这一 LLM 编码高频问题。
> 理论背景：结构化证据单元（Quote–Explanation–Code）常见于可追溯质性分析；**E** 承载推断，**C** 承载可聚合标签。

---

## 1. 一句话区分

| 字段 | 一句话 | 通过测试 |
|------|--------|----------|
| **E (Explanation)** | 这句话**让我们相信什么**（现象、机制、态度、行为）？ | 删掉 Q 之后，E 仍能单独读成一句**可辩驳的主张** |
| **C (Code)** | 在 codebook 里**贴哪个标准标签**？ | 在词表里能**唯一或优先**对应到一条；像「档案柜抽屉名」 |

---

## 2. 自检句（编码时心里念一遍）

1. **E 是否只是在重复 Q？** → 若是，要写出「从 Q 推出的那一步」是什么。
2. **C 读出来就知道发生了什么吗？** → 若读完 C 仍不知道具体行为是什么（如「关联-功能与场景」），说明 C 太泛，需要具体化。参考第 3 节。
3. **同一个 C 能否贴给多篇不相似的 Q？** → 若不能，C 太细或 E 不足；需要改词表或拆 Q。
4. **E 里是否出现「属于某某类」而 C 为空？** → 把「某某类」收成真正的 Code 名写入 C。
5. **C 是否只是 Category 名的复述？** → 如「探询」→ C =「探询-需求」，这是 LLM 常见的泛化错误。C 必须比 Category 更具体一层。

---

## 3. Code 的具体性标准

> 本节来自实践经验：LLM 编码（以及缺乏训练的编码员）倾向于生成**中等抽象度**的 Code——比 Category 窄一点，但读完后仍不知道具体发生了什么。这类 Code 在聚合时毫无区分力，是最常见的隐性质量问题。

### 判断标准

**好的 Code**：读 Code 就知道发生了什么，不需要回看原文。

| 评级 | 例子 | 说明 |
|------|------|------|
| 好 | `关联-快充与开会补电` | 明确的动作 + 具体场景 |
| 好 | `异议处理-以旧换新抵价差` | 明确的策略 + 触发条件 |
| 差（泛化） | `关联-功能与场景` | 等于没说，任何「关联」都能贴 |
| 差（泛化） | `探询-需求` | 只是把 Category 复述了一遍 |
| 差（过长） | `通过提问了解客户对电池续航的具体需求和使用场景` | 这是 E，不是 C |

### 推荐格式

`{动作}-{具体对象/场景}`，通常 3–8 个汉字。具体性优先于长度限制。

### LLM 编码警告

当使用 LLM 进行 QEC 编码时，必须特别防范以下泛化模式：

- **Category 复述**：C 直接使用 Category 名（如「产品呈现」），未往下具体化
- **万能中间层**：如「功能与场景」「需求与痛点」——看起来有信息量，实际能匹配 80% 的 Q
- **形式替代内容**：如「开放式提问」「封闭式提问」——描述的是提问形式，而非行为意图

**建议**：在编码 prompt 中明确给出好/差 Code 对比例（至少 3 对），并在输出中检查唯一率（目标 >90%）。若唯一率过低，说明 Code 泛化严重。

## 4. 正例（结构示意）

| Q（节选） | E | C |
|-----------|---|---|
| 「我先问他平时拍照多不多……」 | 销售通过**提问**收集使用场景信息，为后续匹配机型做依据 | `探询-拍照频率` |
| 「差 300 块我帮他从以旧换新找回来」 | 面对价格敏感，通过**组合权益**维持成交价感知，而非单纯降价 | `异议处理-以旧换新抵价差` |

---

## 5. 反例与改法

| 问题 | 原样 | 改法 |
|------|------|------|
| C 里写满机制 | C = 「通过提问了解需求再推荐」 | C 收窄为词表标签；机制全文放 E |
| E 只有「客户提到价格」 | 过薄 | E 补全：「提及价格」**意味着什么**（敏感/比价/预算硬上限） |
| E 与 C 完全同义 | E = 「价格异议」，C = 「价格异议」 | E 写**如何回应或证据强度**；C 保持类别名 |
| C 是泛化标签 | C = 「关联-功能与场景」 | C 必须具体到读出即知行为：改为如 `关联-快充与通勤` |

---

## 6. 与其他数据源的特别提示

- **对话数据（通用）**：E 宜写「**该轮在对话功能上承担的角色**」（信息收集 / 推进成交 / 修复关系），避免复述说话内容。
- **对话数据（上下文依赖型）**：E 还应**能锚定到前序发言**（上下文依赖），说明「此话在回应对方的哪类刺激」；仅写功能角色仍可能不足。同一轮次可对应 **零到多个 C**（多条 QEC），须在 codebook 约定统计方式。
- **评价文本**：一条评价可拆多条 QEC；**一条 Q 不宜混合两个独立抱怨**。
- **高分样本子集**：E 中可注明「该条来自高分筛选子样本」，并说明筛选门槛的方法论含义（如态度与能力双重过滤），便于 Cross-stream 对照。

---

## 7. 需要升级讨论时

若团队对同一批 Q 的 E/C 仍争执不下：

1. 先检查 **codebook 该 Code 的包含/排除** 是否写清；
2. 仍不行 → 开一条 **ADR**，允许暂时双码或新增第三码，并设合并截止日。
