# IPP 研究方法论工具包

欢迎。这里是一套可复用的 **质性文本研究协议** 模板，与具体行业、具体数据集解耦。你在做访谈编码、对话分析、评价文本、工单等 **需要「可追溯的结构化单元 + 主题聚合」** 的研究时，可以按下面顺序使用本目录中的文件。

本工具包在 **GMBA IPP** 项目中与真实研究并行演进，核心方法见各 Stream 的详细文档（见文末链接）。

---

## 这套工具解决什么问题

1. **统一语言**：用 **QEC（Quote–Explanation–Code）** 把「原文证据 → 你的解释 → 可统计的标签」拆开，避免编码时混在一起。  
2. **少踩坑**：尤其是 **E（解释）和 C（代码标签）** 的分工——详见 `E_vs_C.md`。  
3. **换数据也能起步**：新数据源先填 `domain_adaptation_intake.md`，再决定编码方案和流水线。  
4. **决策留痕**：重要方法选择用 `ADR_TEMPLATE.md` 记在 `decisions/` 里，方便以后写论文、答辩或交接。

## 通用流程与关键决策点

无论数据是深度访谈、模拟对话、平台评价还是工单日志，QEC 质性分析的整体框架一致：

```
Step 0  数据准备与探索
Step 1  编码体系设计与预测试   ← 决策点 1: E/C 操作定义适配
Step 2  全量 QEC 编码           ← 决策点 2: Code 粒度控制
Step 3  主题聚类                ← 决策点 3: 聚类路径选择
Step 4  产出与报告
Step 5  跨源交叉验证（可选）
```

每个决策点都需要**根据你的数据形态做出具体选择**，并在 codebook 或 ADR 中留痕。以下逐一说明。

---

### 决策点 1: E/C 操作定义适配（Step 1）

QEC 框架统一——Q = 原文证据、E = 研究者解释、C = 可聚合标签——但 **E 和 C 的具体操作定义需要按数据类型调整**。这不是偏离标准，而是方法设计的一部分。

| 数据类型 | E 需要额外做什么 | C 有什么特殊约束 |
|----------|-----------------|-----------------|
| 深度访谈 | 说明受访者声称的动机或态度；注意「声称 ≠ 实际」 | 通常一个 QEC 单位对应一个 Code |
| 对话/交互 | 必须锚定前序发言作为上下文（「此话在回应什么」） | 一轮可产出多个 Code（多条 QEC） |
| 评价文本 | 捕捉评价维度与情绪强度 | 一条评价可拆多条 QEC |
| 工单/日志 | 重建事件流和因果关系 | 注意时间戳关联 |

**动作**：在 codebook 的「E 与 C 边界」节（`CODEBOOK_TEMPLATE.md` 第 7 节）中，明确写出本项目对 E 和 C 的特殊约定。重大调整记 ADR。

> 参考脚注：对话数据 E/C 适配的详细论证见 `E_vs_C.md` 第 6 节。

---

### 决策点 2: Code 粒度控制（Step 1–2）

Code 的具体程度**直接影响下游聚类质量**。

- **具体 Code**（推荐）：读 Code 就知道发生了什么，如 `异议处理-以旧换新抵价差`。
  - 下游效果：BERTopic 自然产出均衡主题，单路聚类即可。
- **泛化 Code**：读完后仍不知道具体行为，如 `关联-功能与场景`。
  - 下游效果：大量 Code 聚成一个大团，需要双路聚类做二次拆分。

**判断标准**：唯一率（唯一 Code 数 / 总 Code 数）> 90% 为健康。低于此值说明存在系统性泛化。

**LLM 编码警告**：LLM（及缺乏训练的编码员）倾向于生成中等抽象度的 Code——看起来有信息量，但聚合时毫无区分力。在 prompt 中必须给出好/差 Code 对比例（至少 3 对）。

详见 `E_vs_C.md` 第 3 节「Code 的具体性标准」。

---

### 决策点 3: 聚类路径选择（Step 3）

Code 粒度决定了你应该走单路还是双路聚类：

| 路径 | 适用条件 | 做法 |
|------|----------|------|
| **单路聚类** | Code 粒度高、唯一率 > 90%、BERTopic 无超大主题（> 50% 的 Code 聚入同一主题） | UMAP 降维 → HDBSCAN 聚类 → 层次合并选拐点 |
| **双路聚类** | Code 粒度偏低，或 BERTopic 产出超大主题 | 路径 A：对非超大主题直接提取；路径 B：对超大主题重新 embedding + 聚类，再与路径 A 结果合并 |

**建议**：优先在 Step 2 提升 Code 具体性（治本）；双路聚类是下游补救（治标）。

**必须记录在 ADR 中的内容**：聚类模型及版本、参数（n_neighbors, min_cluster_size 等）、合并层级选择及理由、主题可信度评估标准。

> 参考脚注：已跑通的单路聚类案例见 `stream1a/`，双路聚类案例见 `stream2-analysis/`。

---

### 迭代编码（可选路径）

若数据类型较新（如模拟对话、客服日志），初始编码方案可能不适配。此时可采用迭代路径：

1. **第一轮（形式/结构编码）**：验证数据可分析性，发现编码方案与数据形态的错位
2. **预测试**：确认新方案的覆盖率与区分度
3. **第二轮（内容/行为编码）**：用适配后的方案做全量编码

两轮之间必须做预测试，方案变更记录为 ADR。

---

### 建议显式写进 ADR 的决策类型

以下类型的决策容易在事后被遗忘，建议逐一检查并在 ADR 中留痕：

- 样本筛选标准及其方法论含义（如「高分」的双重含义）
- 数据结构问题（如存储顺序 ≠ 时间顺序）
- E/C 操作定义的适配决策
- 哪些一致性检验未做，及未做的理由
- 聚类路径选择及参数
